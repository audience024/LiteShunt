# [POC-002] 验证按应用识别 TCP Flow

## 1. 任务元信息
- 编号：`POC-002`
- 状态：`BLOCKED`
- 优先级：`高`
- 当前阶段：`M2 POC 验证`
- 前置依赖：`POC-001`
- 关联任务：`CFG-001`、`POC-003`、`EXT-001`
- 最后更新时间：`2026-04-23 11:10 CST`
- 最后执行者：`Codex`
- 最后安全检查点：`宿主最小配置保存链路、扩展诊断日志与最小透明代理规则已落代码；当前因签名与 capability 条件不足，真实带签名验证暂停。`

## 2. 任务目标

在真实 `NetworkExtension` POC 链路下，确认 LiteShunt 扩展能够接收到首个目标应用发起的 TCP Flow，并提取出可用于规则匹配的来源应用身份信息。

## 3. 背景

`POC-001` 已证明最小 Xcode 工程、宿主应用入口和透明代理扩展入口可构建。当前最大的真实不确定性已经从“工程能否建起来”转移到“系统是否真的把目标 Flow 送进扩展，以及扩展能否稳定拿到应用身份元数据”。如果这一步不先验证，后续 `SOCKS5` 转发、回环规避和 `FAIL_CLOSED` 都缺乏真实落点。

## 4. 范围

- 允许修改的目录：
  - `LiteShuntApp/`
  - `LiteShuntExtension/`
  - `LiteShuntShared/`
  - `LiteShuntTests/`
  - `Tests/`
  - `script/`
  - `docs/ai/`
- 允许新增的最小必要内容：
  - 扩展中的 Flow 元数据采集与诊断日志
  - 宿主侧透明代理配置保存 / 加载的 POC 级打通
  - 用于验证来源应用识别的最小测试或人工验证脚本
  - 与本任务对应的交接和验收记录

## 5. 非目标

- 不实现完整 `SOCKS5` 转发链路
- 不扩展到 `UDP`、`QUIC`、全量 `DNS`
- 不实现完整菜单栏 UI 或规则管理页面
- 不在本轮完成回环规避和 `FAIL_CLOSED` 的最终系统级联调
- 不为了调试方便把命中流量偷偷降级为直连

## 6. 前置条件

1. `POC-001` 的工程骨架、宿主入口和扩展入口处于可构建状态。
2. 本机具备 `NetworkExtension` 基础开发条件，且存在可用于透明代理扩展调试的付费开发团队与对应能力配置。
3. 共享配置模型的现有设计不与 `TECHNICAL_DESIGN.md` 冲突。

## 7. 完成标准

1. 宿主侧存在可用于保存或加载透明代理配置的最小 POC 链路，且关键阻塞点有明确记录。
2. 扩展中可记录首个 TCP Flow 的来源应用身份信息，至少覆盖一个命中应用样本。
3. 能明确区分“命中应用有进入扩展”和“未命中或无关流量未被错误识别”的验证结果或已知限制。
4. 相关验证步骤、命令、结果和阻塞项被写入 `HANDOFF.md` / `QA_REPORT.md`，便于后续接手。
5. 未引入超出当前阶段边界的代理能力或 UI 范围扩张。

## 8. 验收方式

1. 执行 `swift build`、`swift test`、`swift run LiteShuntSmokeTests`，确认现有基础链路未被破坏。
2. 执行与工程相关的验证命令，例如 `xcodebuild -list -project LiteShunt.xcodeproj` 和本轮需要的构建 / 运行脚本。
3. 检查扩展日志或诊断记录，确认存在真实 Flow 进入扩展的证据，以及对应的来源应用身份信息。
4. 核对 `docs/ai/HANDOFF.md` 与 `docs/ai/QA_REPORT.md`，确认阻塞点、限制和下一步建议清晰可追溯。
