# LiteShunt Constitution（项目宪章）

## Core Principles（核心原则）

### I. Product Boundary First（产品边界优先）
LiteShunt 的定位始终是“按应用识别 + 规则匹配 + 流量转发编排”的轻量工具，而不是代理软件本体，也不是 `Clash Verge` 的替代品。

约束：
- 仅支持 `macOS 14+`
- 第一版仅支持 `TCP`
- 第一版仅对接本机 `Clash Verge`
- 不实现节点订阅、策略组管理、规则引擎 UI、通用网络调试台
- 不擅自扩展到 `UDP`、`QUIC`、完整 `DNS` 接管，除非技术设计文档明确升级范围

所有规格、计划和任务都必须先证明“没有越出产品边界”，再讨论实现细节。

### II. Official APIs and Lightweight Architecture（官方能力与轻量架构）
LiteShunt 必须优先建立在 Apple 官方、可持续维护的系统能力之上，并保持宿主应用、扩展与共享模块的物理分层和职责清晰。

约束：
- 透明代理主线固定为 `NetworkExtension / NETransparentProxyProvider`
- 不以 `PF` 劫持、非官方 API 或自研代理核心作为主线方案
- 不引入额外常驻后台进程承担宿主应用与扩展通信
- 宿主应用、扩展、共享模型必须分层，不允许长期职责混写
- 新增复杂抽象前，必须先证明现有轻量结构无法满足需求

任何实现如果提升了复杂度，却不能明显降低风险或增加可验证价值，应默认拒绝。

### III. Safety and Traffic Integrity（安全性与流量一致性）
命中规则的流量行为必须稳定、可预期、不可静默泄漏。代理不可用时，行为一致性优先于“看起来可用”。

约束：
- 命中规则的应用默认执行 `FAIL_CLOSED`
- 不允许在代理异常时私自降级为直连
- 必须始终做回环规避，至少排除：
  - LiteShunt 宿主应用
  - LiteShunt 扩展自身
  - `Clash Verge` 及相关核心进程
- 配置快照解析失败时必须保留上一版有效配置
- 日志必须分级，默认不输出完整流量内容、敏感地址或过量调试信息

所有涉及连接失败、回环风险、配置损坏和后端不可达的设计，都必须明确写出失败路径和保护措施。

### IV. Versioned Shared Configuration（可版本化共享配置）
宿主应用与扩展之间的协作必须围绕“可版本化、可整体替换、可安全回退”的配置快照展开，避免引入隐式状态和分散默认值。

约束：
- 共享配置采用 `App Group + JSON 配置快照`
- 配置模型必须支持版本号、整体替换和失败回退
- 默认值、错误码、超时、端口、失败策略等必须集中定义，禁止魔法常量
- 共享模型、常量、错误定义和通用工具统一放在共享模块中
- 规则匹配优先围绕应用身份信息，例如 `Bundle ID`、签名标识等

所有新增配置字段都必须回答三个问题：默认值是什么、如何回退、宿主与扩展如何保持一致。

### V. Verification Before Expansion（先验证，再扩展）
LiteShunt 处于 POC 向 v1 演进阶段，所有核心能力都必须先被验证，再进入更广范围的功能扩展。

约束：
- 核心逻辑必须具备单元测试
- 至少覆盖：
  - 应用规则匹配
  - 排除名单匹配
  - 配置快照解析
  - `SOCKS5` 建链流程
  - 失败策略执行
- 无法通过纯单元测试覆盖的 `NetworkExtension` 行为，必须通过 POC 或集成验证补充
- 在声明任务完成前，应至少完成一次与当前阶段相匹配的构建或验证

若真实系统验证尚未成立，不得用更大范围的功能开发掩盖核心链路的不确定性。

## Non-Negotiable Constraints（不可妥协约束）

### 技术与平台
- 语言与工程主线：`Swift 6+`
- UI 主线：`SwiftUI`，仅在必要时引入 `AppKit`
- 系统能力：`NetworkExtension`
- 测试体系：`XCTest`
- 默认用户可见文案：简体中文

### 实现红线
- 不把 LiteShunt 做成完整代理客户端
- 不引入节点配置、订阅管理、策略组等产品能力
- 不为了省事引入未经设计确认的第三方网络库或代理内核
- 不在没有测试或验证说明的前提下引入复杂并发逻辑
- 不允许多个核心类型长期堆放在同一个职责混杂的文件中

### 默认值
- 默认代理地址：`127.0.0.1`
- 默认代理端口：`7890`
- 默认协议：`SOCKS5`
- 默认连接超时：`3000ms`
- 默认失败策略：`FAIL_CLOSED`

## Development Workflow and Quality Gates（开发流程与质量门禁）

### 1. 开始任何新功能前
- 必须先核对 [TECHNICAL_DESIGN.md](/Users/zero/Downloads/code/LiteShunt/TECHNICAL_DESIGN.md)、[PLAN.md](/Users/zero/Downloads/code/LiteShunt/PLAN.md) 和 [AGENTS.md](/Users/zero/Downloads/code/LiteShunt/AGENTS.md)
- 必须确认当前里程碑边界，避免跨阶段抢跑
- 必须先判断该工作属于：
  - POC 验证
  - 共享配置与模型完善
  - 宿主应用能力建设
  - 扩展转发链路建设
  - 测试与诊断完善

### 2. 进入 spec-kit 工作流时
- `spec.md` 必须明确范围内目标与范围外内容
- `plan.md` 必须解释为何符合本宪章，尤其是：
  - 是否保持轻量架构
  - 是否坚持官方 API 主线
  - 是否处理 `FAIL_CLOSED` 与回环规避
  - 是否明确配置默认值、回退与共享边界
- `tasks.md` 必须把验证任务和实现任务同时落地，不能只写功能开发不写验证

### 3. 完成实现前
- 至少补齐与改动相匹配的测试或验证说明
- 至少执行一次可行的构建、测试或 POC 验证
- 如涉及系统权限、签名、扩展生命周期、回环规避、失败策略，必须补中文注释说明边界与风险
- 如新增命令、目录结构、测试策略，应同步更新相关文档

### 4. 当前阶段优先级
在 `POC-002` 阶段及其相邻任务中，优先级默认遵循：
1. 真实 Flow 捕获与来源应用识别
2. `SOCKS5` 转发链路成立
3. 回环规避与 `FAIL_CLOSED` 验证
4. 共享配置热加载与宿主应用交互完善
5. UI 丰富化与体验补充

## Governance（治理）

本宪章用于约束 LiteShunt 的 spec、plan、tasks 和实现决策，是后续规格驱动开发的基线文件。

治理规则：
- 所有新规格、实现计划和任务拆分都必须检查是否符合本宪章
- 若某项设计违反本宪章，必须在计划中显式说明原因、风险、替代方案以及为何仍要继续
- 若现有项目文档之间出现冲突，默认先按既有项目规则处理：
  - `TECHNICAL_DESIGN.md` 优先于 `PLAN.md`
  - `PLAN.md` 优先于 `AGENTS.md`
  - 宪章应在人工确认后同步修订，以反映最终共识
- 任何扩大产品边界、改变默认失败策略、放松回环规避要求、引入新的系统主线能力的变更，都必须同时更新：
  - 技术设计文档
  - 任务计划文档
  - 本宪章
